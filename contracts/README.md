# Recurra Contracts

Foundry project — all three contracts complete: `SubscriptionRegistry`,
`SubscriptionVault`, `PaymentExecutor` (+ `MockUSDC` test token in `script/`).

```shell
forge build   # compile
forge test    # run the suites (registry, vault, executor, integration)
```

## Local dev loop (the M2 harness)

Terminal 1 — a local chain:

```shell
anvil
```

Terminal 2 — deploy + wire everything (anvil's prefunded account #0):

```shell
PRIVATE_KEY=0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80 \
forge script script/Deploy.s.sol --rpc-url http://127.0.0.1:8545 --broadcast
```

One script, atomic wiring: Registry → Vault → Executor → `setExecutor` on
both → `setAuthorizedExecutor` → MockUSDC. `AUTHORIZED_EXECUTOR` env picks
the scheduler wallet (the Openfort address at M3); unset, it defaults to the
deployer, so one anvil key runs everything locally — the backend's local
signer is simply account #0.

Fresh-anvil addresses are deterministic; backend `.env`:

```
ARBITRUM_RPC=http://127.0.0.1:8545
SUBSCRIPTION_REGISTRY_ADDRESS=0x5FbDB2315678afecb367f032d93F642f64180aa3
SUBSCRIPTION_VAULT_ADDRESS=0xe7f1725E7734CE288F8367e1Bb143E90bb3F0512
EXECUTOR_ADDRESS=0x9fE46736679d2D9a65F0992F2272dE9f3c7fa6e0
# MockUSDC: 0x0165878A594ca255338adfa4d48449f69242Eb8F
```

Restarting anvil wipes the chain; rerun the deploy, same addresses.

## Seeding a full payment flow

Anvil default accounts: #0 deployer/scheduler, #1 merchant, #2 subscriber.

```shell
RPC=http://127.0.0.1:8545
REG=0x5FbDB2315678afecb367f032d93F642f64180aa3
VAULT=0xe7f1725E7734CE288F8367e1Bb143E90bb3F0512
EXEC=0x9fE46736679d2D9a65F0992F2272dE9f3c7fa6e0
USDC=0x0165878A594ca255338adfa4d48449f69242Eb8F
MKEY=0x59c6995e998f97a5a0044966f0945389dc9e86dae88c7a8412f4603b6b78690d   # merchant
AKEY=0x5de4111afa1a4b94908f83103eb1f1706367c2e68ca870fc3fb9a804cdab365a   # subscriber
SKEY=0xac0974bec39a17e36ba4a6b4d238ff944bacb478cbed5efcae784d7bf4f2ff80   # scheduler
ALICE=0x3C44CdDdB6a900fa2b585dd299e03d12FA4293BC

# merchant: "10 USDC / 30 days"
cast send $REG "createPlan(address,uint256,uint256)" $USDC 10000000 2592000 --private-key $MKEY --rpc-url $RPC
# subscriber: subscribe (plan 1), mint + approve + escrow 60 USDC
cast send $REG "subscribe(uint256)" 1 --private-key $AKEY --rpc-url $RPC
cast send $USDC "mint(address,uint256)" $ALICE 60000000 --private-key $AKEY --rpc-url $RPC
cast send $USDC "approve(address,uint256)" $VAULT 60000000 --private-key $AKEY --rpc-url $RPC
cast send $VAULT "deposit(address,uint256)" $USDC 60000000 --private-key $AKEY --rpc-url $RPC

# the scheduler's job (what the backend automates): charge sub 1
cast send $EXEC "executePayment(uint256)" 1 --private-key $SKEY --rpc-url $RPC

# verify
cast call $REG "isDue(uint256)(bool)" 1 --rpc-url $RPC                      # false
cast call $VAULT "balances(address,address)(uint256)" $ALICE $USDC --rpc-url $RPC  # 50000000
```

A second `executePayment` immediately reverts `NotDue()` — the benign
scheduler race. To make the sub due again without waiting 30 days:

```shell
cast rpc evm_increaseTime 2592000 --rpc-url $RPC && cast rpc evm_mine --rpc-url $RPC
```

From M3, deployed addresses live in `deployments.md` at the repo root — env
files only, never hardcoded.
